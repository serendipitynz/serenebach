# レンタルサーバで公開するガイド（CGI）

> **これは素案（ドラフト）です。** スクリーンショットは未挿入、サーバ別の手順は**さくらのレンタルサーバを実例**として先行整備し、エックスサーバー / ロリポップは「まず確認すべきこと」を示す段階です（後述）。

このガイドは、[Serene Bach 2 からの移行ガイド](sb2-migration.ja.md) などでパソコンの中まで用意できた新しい Serene Bach を、**いま使っているレンタルサーバに置いて公開する**ための手引きです。FTP とブラウザを中心に説明します。

新しい Serene Bach は **1つの実行ファイル**なので、Perl 版のように多数のファイルを設置する必要はありません。ここでは、いまの Serene Bach 2 と同じ「CGI」として動かす方法を説明します。

## この方法が使えるサーバ（最初に確認）

⚠️ **最重要:** この方法は「**自分で用意したプログラム（コンパイル済みバイナリ）を CGI として実行できる**」レンタルサーバでのみ使えます。多くの共有レンタルサーバは Perl や PHP は動かせても、**任意のバイナリの実行を許可していない**ことがあります。

- **さくらのレンタルサーバ**：実行できます（このガイドの実例）。
- **エックスサーバー / ロリポップ など**：**要確認**。契約プランで「独自 CGI（バイナリ）の実行」が許可されているかを、各社のヘルプやサポートで確認してください（[サーバ別メモ](#サーバ別メモ)）。

動かせない場合は、常駐サーバ（VPS）や、静的書き出し（[deployment.md](../deployment.md) の Static / Hybrid）などの別の方法を検討します。

---

## 用意するもの

FTP で次の 3 点をサーバへ置きます。

1. **サーバ用のバイナリ** … サーバの OS に合ったもの（例：さくら＝`freebsd` / `amd64`）。⚠️ 作業用パソコンの `serenebach.exe`（Windows）などは**サーバでは動きません**。入手先は[移行ガイドのステップ2-1](sb2-migration.ja.md#ステップ2新しい-serene-bach-を手に入れる)。
2. **データベース** `data/dev.db` … 移行で取り込んだ記事・コメントが入ったファイル。新規に始める場合は不要（サーバ上の初回アクセスで作られます）。
3. **`.htaccess`** … ブログの URL を、このプログラムに渡すための設定ファイル。

---

## 共通の手順

1. **バイナリをアップロード**：サーバ用バイナリを、ブログを置きたい場所（例：`~/www/blog/`）へ FTP でアップロードし、**`serenebach.cgi` にリネーム**します。
2. **実行権限を付ける**：FTP ソフトの「パーミッション（属性）変更」で `serenebach.cgi` を **755** にします。
3. **データベースを置く**：`serenebach.cgi` と同じ場所に `data/` フォルダを作り、その中に `dev.db` を置きます（＝`data/dev.db`）。移行データを引き継ぐ場合は、作業用パソコンの `data/dev.db` をアップロードします。⚠️ この DB には管理者パスワードのハッシュやコメント情報が入っています。**後述の `.htaccess` で `data/` を必ず遮断**し、URL から直接ダウンロードされないようにしてください。
4. **`.htaccess` を置く**：下記の内容の `.htaccess` を同じ場所に置きます。
5. **ブラウザで開く**：ブログの URL を開きます。移行データを載せた場合は管理者がすでにあるので、`/admin/login` からログインできます。新規の場合は自動的に `/setup` に移り、管理者を作成できます。

> 💡 テキストファイル（`.htaccess`）はアップロード時に**改行コードや文字コード**の扱いに注意してください（多くの FTP ソフトは自動処理します）。

## `.htaccess` の書き方

Serene Bach は、リクエストされたパス（例：`/entry/5/`）が CGI に正しく渡れば動きます。Apache では次のように振り分けます。**順番が重要**です（① DB を遮断 → ② 公開してよい静的ファイルだけ直接配信 → ③ 残りはすべて CGI）。

```apache
RewriteEngine On
# ブログをサブパス（例 /blog/）に置く場合は、実際のパスに合わせる:
RewriteBase /

# ① データベースなど内部ファイルを外部から遮断する（最優先・必須）
RewriteRule ^data/ - [F]

# ② すでにサーバにある「古い画像ファイル」だけ直接配信する。
#    実在チェック(-f)で絞るのがポイント。存在しない画像リクエストは③で
#    Serene Bach に渡り、アップロード画像・OG カードなどが配信される。
RewriteCond %{REQUEST_FILENAME} -f
RewriteRule ^img/ - [L]

# ③ それ以外はすべて CGI へ渡す（ブログ本体・旧 sb.cgi・旧 *.html・
#    カテゴリーディレクトリも Serene Bach に届き、旧 URL 転送が働く）
RewriteRule ^(.*)$ serenebach.cgi/$1 [L]
```

- **① の DB 遮断は必ず入れてください。** これが無いと、`/blog/data/dev.db`（管理者パスワードのハッシュやコメントの情報を含む SQLite）が URL から直接ダウンロードできてしまいます。より確実にしたい場合は、DB をドキュメントルート外に置き、環境変数 `SB_DB` でその場所を指定します（上級者向け）。
- **② は「実在する画像ファイルだけ」直接配信します。** `%{REQUEST_FILENAME} -f` で存在チェックしているため、Serene Bach がアプリ側で配信する画像（新しくアップロードした画像・OG カード画像など、ディスク上に実体が無いもの）は③で CGI に渡り、正しく表示されます。逆に **`-f` を外して `^img/` を一律素通しにするとこれらが 404 になる**ので注意してください。
- 実在する古い静的ファイル（`log/eid2.html` など）やカテゴリーディレクトリも、②で素通しにせず**③で CGI に渡す**のがポイントです。こうすることで、移行ガイドで説明した**昔の URL の転送**（`log/eid2.html` → `/entry/{key}/` など）が Serene Bach に届いて機能します。
- ⚠️ `-f` ／ `-d` で「実在ファイル・ディレクトリを**一律に**素通し」する書き方は避けてください（DB 流出や旧 URL 転送の失敗につながります）。素通しは②のように**パスを限定**し、可能なら実在チェックも付けます。
- ブログを `https://example.com/blog/` のようにサブパスへ置く場合は、`RewriteBase /blog/` にし、`.htaccess` はその `blog/` フォルダに置きます。

> ⚠️ `.htaccess` の許可範囲（`RewriteEngine` や CGI 実行の可否）、DB を確実に隠す方法はサーバによって異なります。上記は一般的な例です。**確定情報は [deployment.md の CGI 節](../deployment.md) と、各サーバのヘルプで確認してください。**

## 管理画面を速くする（任意）

CGI では 1 リクエストごとにプログラムが起動するため、管理画面の CSS/JS 読み込みが毎回少し重くなります。次の 2 つで軽くできます（詳細は [deployment.md](../deployment.md)）。

1. **ETag / 304**：初回以降はブラウザキャッシュが効きます（設定不要）。
2. **`extract-assets`**：管理画面の静的ファイルをディスクに書き出し、`.htaccess` で直接配信します。

```bash
./serenebach extract-assets --out=./admin-static
```

書き出したら、`.htaccess` の **③（CGI へ渡すルール）の前**に次を足します。存在チェックの `%{DOCUMENT_ROOT}/…/admin-static/$1` は、**設置パスを含めて**指定するのが要点です（ここが実際の場所とずれると直接配信が効きません）。

```apache
# ルート直下（https://example.com/）に設置した場合:
RewriteCond %{DOCUMENT_ROOT}/admin-static/$1 -f
RewriteRule ^admin/static/(.+)$ admin-static/$1 [L]

# サブパス（例 https://example.com/blog/）に設置した場合は、設置パスを含める:
# RewriteCond %{DOCUMENT_ROOT}/blog/admin-static/$1 -f
# RewriteRule ^admin/static/(.+)$ admin-static/$1 [L]
```

存在チェックで絞っているので、書き出していないファイルは③で CGI に渡り、そのまま配信されます。バイナリを更新するたびに `extract-assets` を再実行してください（ETag が変わるため）。確定的な書き方は [deployment.md の CGI 節](../deployment.md) を参照してください。

## サーバ別メモ

> このセクションは整備中です。**さくらのレンタルサーバ**を先行して実例化し、他社は「確認すべきこと」を挙げています。実機で確認でき次第、手順を追記します。

### さくらのレンタルサーバ

- OS / CPU：**FreeBSD / amd64**（バイナリは `freebsd` / `amd64` を選ぶ）。[動作環境](https://help.sakura.ad.jp/rs/2251/)。
- 独自 CGI（バイナリ）の実行：可能。
- 手順：上の[共通の手順](#共通の手順)どおり。CGI を置く場所・実行権限（755）・`.htaccess` の書き方はさくらのヘルプに沿って調整。

### エックスサーバー / ロリポップ など

- **まず確認**：契約プランで「独自 CGI（自分でアップロードしたバイナリ）の実行」が許可されているか。許可されていなければ、この方法は使えません。
- OS は多くが **Linux / amd64**（バイナリは `linux` / `amd64`）。
- CGI を置けるディレクトリ、`.htaccess` で `RewriteEngine` が使えるか、実行権限の付け方は各社で異なります。**各社のヘルプで確認が必要です。**

## 困ったときは

| 症状 | 確認すること |
| --- | --- |
| 500 Internal Server Error | `serenebach.cgi` の**実行権限（755）**、ファイル名（`.cgi`）、`.htaccess` の内容。サーバのエラーログも確認。 |
| そのバイナリが動かない | **サーバの OS / CPU 用**のバイナリか（Windows の `.exe` をサーバに置いても動きません）。さくら＝FreeBSD、他社の多くは Linux。 |
| ページは出るが URL がおかしい | `.htaccess` の `RewriteBase` が実際の設置パスと合っているか。サブパス設置なら `/blog/` などに。 |
| 管理画面の見た目が崩れる | `extract-assets` を使う場合、バイナリ更新のたびに再実行が必要（ETag が変わるため）。 |
| 昔のリンクが転送されない | `configure.cgi` を含めて移行したか（[移行ガイド](sb2-migration.ja.md)）。`.htaccess` で旧 URL が CGI に渡っているか。 |
